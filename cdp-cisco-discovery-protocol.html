CDP – Cisco Discovery Protocol

Cisco Discovery Protocol – proprietary Layer 2 network protocol for device discovery, topology mapping, and neighbour identification

Overview

Cisco Discovery Protocol (CDP) is a proprietary Data Link Layer (Layer 2) protocol developed by Cisco Systems. It enables Cisco network devices — routers, switches, access points, and IP phones — to automatically discover and share information about directly connected neighbours. CDP operates independently of network layer protocols and functions even when IP is not configured, making it a foundational tool for network visibility, topology documentation, and troubleshooting.

Key Value: CDP provides automatic, real-time visibility into directly connected Cisco devices — including device type, IOS version, IP address, and interface details — without requiring manual inventory, making it indispensable for network documentation, troubleshooting, and VoIP deployments.

How CDP Works

Protocol Mechanics

  • CDP operates at Layer 2 (Data Link) using multicast MAC address 01:00:0C:CC:CC:CC
  • Devices send CDP advertisements every 60 seconds by default (configurable)
  • Advertisements are received and stored in the CDP neighbour table for 180 seconds (hold time)
  • CDP is enabled globally and per-interface on Cisco devices by default

Information Advertised

  • Device ID — hostname of the neighbour device
  • Local and remote port/interface identifiers
  • Platform and device model (e.g. Cisco Catalyst 9300)
  • IOS/software version running on the neighbour device
  • IP address(es) of the neighbour device
  • VTP domain name and native VLAN information
  • Duplex setting and Power over Ethernet (PoE) capabilities

Key CDP Commands

  • show cdp neighbors – summarised view of directly connected CDP neighbours
  • show cdp neighbors detail – full detail including IP addresses and software versions
  • show cdp interface – CDP status and timer settings per interface
  • no cdp run – disable CDP globally on the device
  • no cdp enable – disable CDP on a specific interface

CDP Use Cases

Network Topology Discovery & Documentation

  • Automatically map physical network topology without manual surveys
  • Identify device interconnections, interface pairings, and uplink paths
  • Support network management tools (e.g. SolarWinds, PRTG, Cisco DNA Center) in auto-discovering topology

VoIP & IP Phone Deployment

  • CDP enables Cisco IP phones to automatically receive VLAN configuration from connected switches
  • Switches advertise voice VLAN information to IP phones via CDP, eliminating manual configuration
  • PoE negotiation between switches and IP phones is facilitated through CDP exchanges

Troubleshooting & Diagnostics

  • Identify misconfigured duplex or speed settings between neighbours
  • Verify correct VLAN assignments and native VLAN consistency across trunk links
  • Confirm physical connectivity and interface state without requiring SNMP or IP reachability

Network Auditing & Compliance

  • Enumerate all Cisco devices and software versions for patch management planning
  • Identify unauthorised or unexpected devices connected to the network
  • Cross-reference CDP data with CMDB records for configuration drift detection

CDP Security Considerations

Information Disclosure Risks

  • CDP advertisements contain sensitive data — device type, IOS version, and IP addresses — that can aid attackers in reconnaissance
  • An attacker with access to a network segment can passively capture CDP frames and map the infrastructure
  • CDP data can be used to identify vulnerable IOS versions targeted for exploitation

CDP Spoofing & Flooding Attacks

  • Malicious actors can send crafted CDP packets to overload the CDP neighbour table (CDP flooding)
  • CDP spoofing can be used to manipulate VoIP phone VLAN assignments
  • Cisco switches are vulnerable to memory exhaustion if flooded with forged CDP advertisements

Hardening Recommendations

  • Disable CDP globally (no cdp run) on devices where neighbour discovery is not required
  • Disable CDP on all external-facing, untrusted, and user-access interfaces (no cdp enable)
  • Retain CDP only on trunk links, uplinks, and infrastructure interconnects where it provides operational value
  • Consider LLDP (Link Layer Discovery Protocol) as an open-standard alternative where vendor-agnostic discovery is needed

CDP vs. LLDP

  • CDP is Cisco-proprietary; LLDP (IEEE 802.1AB) is the open-standard equivalent supported by multi-vendor devices
  • LLDP-MED extends LLDP with VoIP-specific capabilities similar to CDP's IP phone support
  • In mixed-vendor environments, LLDP is preferred for interoperability; CDP remains dominant in all-Cisco deployments
  • Both protocols can run simultaneously on Cisco devices — CDP for Cisco-to-Cisco, LLDP for multi-vendor visibility

Key Trends & Best Practices in 2026

Network Automation & Telemetry

  • Ingest CDP neighbour data into network automation platforms (Cisco DNA Center, Ansible, NetBox) for dynamic topology management
  • Use CDP data as an input to network digital twins for real-time infrastructure modelling
  • Combine CDP with streaming telemetry for comprehensive, up-to-date network visibility

Zero Trust Network Segmentation

  • Audit CDP-enabled interfaces as part of Zero Trust network segmentation reviews
  • Ensure CDP is disabled on all endpoints, guest networks, and untrusted access ports
  • Use CDP topology data to validate and enforce segmentation policies in microsegmented environments

IT Director Responsibilities

  • Ensure CDP security hardening is included in network device build standards and configuration baselines
  • Review CDP enablement as part of regular network security audits and Cyber Essentials assessments
  • Leverage CDP data to maintain accurate network topology documentation and CMDB asset records
  • Coordinate with network engineering teams to establish clear policies on where CDP is permitted vs. disabled
  • Evaluate LLDP as a complementary or replacement protocol in multi-vendor or Zero Trust environments
