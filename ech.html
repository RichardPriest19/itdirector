ECH

Encrypted Client Hello – privacy-preserving TLS extension that hides SNI & handshake metadata

Overview

Encrypted Client Hello (ECH) is a TLS extension (RFC 9257 / draft-ietf-tls-esni) that encrypts the entire ClientHello message — including the Server Name Indication (SNI), ALPN, and other handshake metadata — preventing passive observers from seeing which domain a client is connecting to. In March 2026, ECH is widely deployed and considered **best practice** for public-facing HTTPS services. It is supported by all major browsers (Chrome 117+, Firefox 118+, Safari 16.4+, Edge 117+), most CDNs (Cloudflare, Akamai, Fastly, Google Cloud CDN, AWS CloudFront with custom origins), and a growing number of origin servers (Nginx 1.25+, Apache 2.4.58+, Caddy 2.7+, Traefik, Envoy, HAProxy). ECH is mandated or strongly recommended by privacy-focused standards, privacy regulations (GDPR, CCPA/CPRA, LGPD), cyber insurance policies, zero-trust architectures, and enterprise security baselines to prevent SNI snooping, metadata leakage, and domain-targeted censorship or surveillance. ECH works in conjunction with **TLS 1.3** (mandatory) and **Encrypted DNS** (DoH/DoT) to provide comprehensive connection privacy.

Key Value: ECH significantly enhances user privacy by hiding the destination domain from passive network observers — reducing metadata leakage, preventing domain-targeted attacks & censorship, supporting zero-trust principles, improving compliance with privacy regulations, and — when combined with TLS 1.3, ECH, and DoH — delivering the strongest practical protection for data-in-transit privacy in 2026.

Core Components & Requirements (2026)

ECH Operation

  • Client fetches ECH config from HTTPS DNS record (ECHConfigList)
  • Client encrypts ClientHelloInner & sends encrypted outer ClientHello
  • Server decrypts & processes Inner, responds with encrypted handshake

Key Dependencies

  • TLS 1.3 mandatory (ECH only works over TLS 1.3)
  • Encrypted DNS (DoH/DoT) to protect ECH config fetch
  • Certificate transparency & short-lived certs recommended

Deployment Models

  • CDN-fronted (Cloudflare, Akamai, Fastly) – easiest path
  • Origin server direct (Nginx, Caddy, Envoy, Traefik)
  • Hybrid (CDN + origin fallback)

Privacy & Security Benefits

  • Hides SNI, ALPN, QUIC version, extensions
  • Prevents passive domain fingerprinting & censorship
  • Reduces metadata available to ISPs & middleboxes

Best Practices for ECH Deployment (2026)

Implementation

  • Start with CDN (Cloudflare, Akamai, Fastly) for immediate coverage
  • Enable ECH on origin servers (Nginx, Caddy, Envoy)
  • Use DoH/DoT for ECH config fetch privacy

Security & Privacy Hardening

  • Enforce TLS 1.3 only & strong ciphers
  • Enable HSTS preload & certificate transparency
  • Monitor fallback rates & privacy leakage

Governance & Compliance

  • Conduct periodic privacy impact assessments
  • Align with GDPR, CCPA, DORA & cyber insurance
