OpenID Connect

OpenID Connect (OIDC) – identity layer on top of OAuth 2.x for secure authentication & user info

Overview

OpenID Connect (OIDC) is the modern identity protocol built on OAuth 2.x, adding authentication and user profile information to the authorization framework. In March 2026, OIDC is the de facto standard for secure single sign-on (SSO), identity federation, and user authentication across web, mobile, SPA, native apps, and APIs. It delivers ID tokens (JWTs) containing verified user claims, supports phishing-resistant flows (passkeys/FIDO2/WebAuthn), aligns tightly with OAuth 2.1 (mandatory PKCE, no Implicit/ROPC), and includes advanced profiles like FAPI 2.0 (financial-grade), PAR (Pushed Authorization Requests), JAR (JWT-secured requests), and DPoP (Demonstrable Proof-of-Possession) for token binding. OIDC is implemented by virtually all major identity providers (Entra ID, Okta, Auth0, Google, Ping, Keycloak).

Key Value: OIDC provides secure, interoperable authentication with verified identity claims, enables phishing-resistant SSO, supports passwordless/passkey experiences, and — when combined with OAuth 2.1 — delivers strong authorization + authentication in one protocol flow.

Core Flows & Components (2026)

Primary Flows

  • Authorization Code + PKCE – secure standard flow for all clients
  • Hybrid Flow – code + token for SPAs & mobile
  • Client Credentials – machine-to-machine (no user context)

ID Token & UserInfo

  • ID Token (JWT) – signed, verifiable claims about user & auth event
  • UserInfo endpoint – additional claims via access token
  • Claims: sub, name, email, roles, auth_time, amr (auth method reference)

Security Hardening

  • Mandatory PKCE & state parameter
  • DPoP & sender-constrained tokens
  • PAR & JAR for secure request transmission

Key Features & Trends in 2026

Phishing-Resistant & Passwordless

  • FIDO2/WebAuthn/passkeys as first-class auth methods
  • amr claim values for strong auth (hwk, swk, phr)
  • Integration with FAPI 2.0 for high-security sectors

Zero Trust & Modern Apps

  • Continuous auth signals & step-up authentication
  • Support for SPAs, native apps, & agentic flows
  • Token binding & replay protection

Interoperability & Profiles

  • FAPI 2.0 – financial-grade security baseline
  • Shared Signals Framework (CAEP, RISC) for event notifications
  • Native SSO & federation across IdPs

Best Practices for Enterprise Use (2026)

Secure Implementation

  • Use Authorization Code + PKCE exclusively
  • Enforce DPoP or mTLS for token binding
  • Validate ID token signature & claims (iss, aud, exp, nonce)

Phishing Resistance

  • Require passkeys/FIDO2 for workforce & high-risk apps
  • Deprecate SMS/TOTP as primary MFA
  • Use step-up auth for sensitive actions

Governance & Monitoring

  • Centralize client registration & scope approval
  • Log & audit token issuance/usage
  • Monitor for token misuse & session anomalies

IT Director Responsibilities

  • Define OIDC as primary authentication protocol across apps
  • Approve IdP selection & phishing-resistant requirements
  • Ensure alignment with OAuth 2.1 & Zero Trust standards
  • Oversee client governance, scope minimization & compliance
  • Monitor OIDC adoption, security incidents & user experience
  • Drive migration from legacy flows (Implicit/ROPC) to modern OIDC
  • Plan for emerging profiles (FAPI 2.0, agentic auth)
